# 实体文档与动作文档分层设计

## 1. 文档定位

- 本文档承载 `ent-loom-doc` 针对“实体文档 + 动作文档”的设计基线。
- 它回答“如何对实体及其关联的业务动作进行结构化文档描述，以及最佳实践如何落地”。
- 核心关注文档元数据建模、组装（Assembly）与视图查询。
- 具体的运行时分发、权限治理与执行细节详见各接入组件文档。
- 本文只定义文档核心服务（service/core）能力，不定义固定 Controller 与 HTTP 路径。

## 2. 背景

现代应用通常存在两类元数据：

- 实体元数据：围绕实体类、表名、字段、索引组织。
- 动作元数据：围绕业务操作标识（例如 `entityClass/scene/requestType/responseType`）组织。

典型案例：

- 实体：`Order`（订单）
- 动作：`OrderSubmitAction`（提交订单）
- 请求 DTO：`OrderSubmitRequest`
- 响应 DTO：`OrderSubmitResponse`

业务侧希望能够以实体为中心，聚合展示其关联的所有业务动作文档。

## 3. 当前态问题

### 3.1 现有实体模型只覆盖结构

早期的文档模型（如 `EntityDocCoreService`）通常以“实体”为绝对中心：

- 核心身份是 `resourceCode + entityClass + tableName`。
- 组装逻辑强制要求实体级别的注解支持。
- 这种模型无法自然承载“动作”这类具有过程语义的元数据。

### 3.2 动作元数据在各处碎片化

业务动作的元数据往往散落在执行链注册、Spring Controller 或拦截器配置中，这些信息通常只包含运行时路由（Routing）所需的最少字段，不足以直接形成完整文档。

### 3.3 运行时文档源的稳定性

如果文档过度依赖源码注释（如 JavaDoc），在生产环境下往往无法通过反射获取，导致文档接口输出受限。

## 4. 设计目标

- 保持实体文档的纯粹性，只负责结构化说明。
- 为业务动作类能力新增独立的动作文档（`actionDoc`）模型。
- 文档体系在表现上保持统一，但底层资源模型分离。
- 文档元数据应来自运行时可读取的稳定注解或显式定义。
- 首版支持最小查询与渲染，后续逐步补充权限示例和错误语义。

## 5. 总体结论

最佳实践是：

- **实体文档（entityDoc）** 和 **动作文档（actionDoc）** 分层建设。
- `entityDoc` 回答“这个实体是什么、字段是什么、表结构怎样”。
- `actionDoc` 回答“这个动作怎么调、场景是什么、入参出参是什么”。
- 实体页可以展示动作摘要，但动作详情仍由 `actionDoc` 提供。

## 6. 推荐分层

### 6.1 实体文档 (entityDoc)

职责：
- 实体名称、描述、表名。
- 字段清单、索引清单。
- 字段关联关系。

### 6.2 动作文档 (actionDoc)

职责：
- 动作名称、描述。
- 归属实体与场景标识（Scene）。
- 请求类型与响应类型的结构化文档。
- 调用约束与示例。

## 7. 推荐元数据模型

### 7.1 动作级注解

建议新增动作级文档注解，例如 `@EntDocAction`。
说明：动作注册注解负责运行时分发，`@EntDocAction` 只负责文档语义，实现职责分离。

### 7.2 DTO 级注解

建议为请求/响应 DTO 增加独立注解（如 `@EntDocDto`），便于前端展示模块标题。

### 7.3 参数级注解

建议新增参数级注解（如 `@EntDocParam`）。
说明：不建议直接复用实体字段注解（如 `@EntDocField`），因为 DTO 参数与数据库字段属于不同层次。

## 8. 推荐服务模型

### 8.1 实体文档查询

保留按 `resourceCode` 或 `entityClass` 查询的能力。

### 8.2 动作文档查询

新增 `ActionDocCoreService`。
核心约束：
- `DocActionDefinition` 是核心查询契约。
- 各种来源（如 CRUD 注解、Spring Bean 等）应先转换为该契约。
- 动作唯一键采用 `entityClass + scene`。

## 9. 核心模块拆分建议

- `ent-loom-doc-annotations`：纯粹的文档语义注解。
- `ent-loom-doc-core`：文档模型解析、组装与查询逻辑。

## 10. 组件边界规约

为了保持模块纯粹性，`ent-loom-doc` 严格遵循以下边界：

- **不承接执行逻辑**：文档模块不负责业务请求的实际分发或执行。
- **不定义传输协议**：只提供 Service 层能力，不绑定特定的 HTTP 规范。
- **元数据适配**：与其他组件（如 CRUD）的交互应通过适配层（Adapter）完成。

详细的组件协作规约见：[Ent-Loom 组件边界与协作规约](../../../docs/architecture/core/component-boundaries.md)。
